最近我把 devpiano 推进到了 v0.1.1，这是第一个计划中的发布版本。

写这篇文章不是想做项目介绍——项目介绍看 README 就够了。这篇文章想记录的是：我是怎么从一个 FreePiano 用户，变成独立发起 devpiano 这个开源项目的。

devpiano 是我的第一个从零开始的开源项目。说实话，以前参与过一些项目，但那些都是别人搭的架子、定的方向，自己只是在上面添砖加瓦。devpiano 不一样，从想法到代码仓库，从第一个 commit 到打包发布，全程都是自己在推。这种体验和参与别人的项目完全不同。

## 最初动机

我之前一直在用 FreePiano，一个 Windows 上的键盘钢琴工具。电脑键盘敲下去，钢琴声出来，练琴的时候挺好用的。

后来就想：能不能自己做一个？

FreePiano 是很久以前的项目了，代码是老的 Windows 原生 API 写的，架构也偏老。不是说它不好，只是继续在那个基础上改，感觉不太顺手。

我没有去 fork FreePiano，也没有去跟作者说能不能接手。而是决定自己从头做一个——用现代的框架、现代的工程方式，看看能不能做出一个让自己用起来舒服的东西。

这是 devpiano 的起点：不是要替代 FreePiano，也不是要做一个商业产品，就是一个自己想做的个人项目。

## 为什么是 C++ 和 JUCE

技术选型的时候想了几个选项。

首先是语言。我选了 C++，主要是考虑到音频处理和 MIDI 这一块，C++ 的生态比较成熟，性能也够用。Python 或者其他脚本语言也能做，但总觉得做桌面应用差点意思。C# 也是一个选择，不过我想试试跨平台的可能性，C++ 相对更底层一些。

然后是框架。选 JUCE 是因为它把音频、MIDI、UI 这些我需要的东西都封装好了，不用自己造轮子。JUCE 在音频应用开发里算是比较成熟的框架，社区和文档也还行。

实际用下来，JUCE 的 API 设计确实比直接调 Windows API 舒服很多。音频设备管理、插件宿主、组件树这些，它都帮你抽象好了。当然，JUCE 有自己的学习曲线，但比起自己手写 WASAPI/ASIO 代码，门槛还是低了不少。

开发环境方面，我用的是 WSL 主工作树加 Windows 镜像树的混合模式。代码编辑在 WSL，用 clangd 做 LSP 提示；验证构建在 Windows，用 MSVC。CMake + Ninja 做构建系统，`./scripts/dev.sh` 管理环境自检和构建流程。一开始觉得这套东西挺麻烦的，用熟了之后还挺顺手的。

## 核心功能

说了这么多，devpiano 到底能做什么？

简单来说：电脑键盘可以当钢琴键盘用，按下 QWE 之类的键就能发声。可以加载 VST3 插件，用更专业的音色。可以录音、播放、调速。也可以把演奏保存成文件，以后再打开。

具体一点：

- 电脑键盘映射到 MIDI note，驱动 VST3 插件或者内置的 fallback synth 发声
- 支持外部 MIDI 设备输入
- VST3 插件扫描、加载、编辑
- 演奏录制和回放
- MIDI 文件导入和导出
- WAV 离线导出（用 fallback synth）
- .devpiano 格式保存和打开演奏文件
- 播放速度控制（0.5x 到 2.0x）
- 布局 Preset 保存和切换

这些功能不是一天做出来的，是分了好几个阶段慢慢攒的。

## 从堆功能到可维护

说实话，前面几个阶段我基本是"堆功能"的状态。能跑就行，代码结构没怎么管。MainComponent 越来越长，一个文件写了一千五百多行，自己看着都头疼。

后来觉得这样下去不行，就开始做架构收敛。把 MainComponent 里的逻辑拆分出去：布局管理、录制回放控制、插件操作、设置窗口、状态快照构建，各自独立成模块。

拆分完 MainComponent 从 1587 行降到了 600 行左右。不是说我重构有多厉害，就是把原来堆在一起的代码按职责分开，各管各的，边界清晰一点。

这个过程让我体会到：个人项目开始的时候先跑起来是对的，但到了一定阶段还是要回头整理，不然后面改起来越来越费劲。

## AI Agent 辅助开发

devpiano 开发过程中我用了 AI Agent 辅助。不是那种"AI 全自动写代码"的用法，更像是有一个随时可以问的助手。

AI 能够辅助实现我的想法，这让我觉得很厉害。Agent 能够使用各种工具和技能，帮助我把项目做得越来越标准化、工程化，这个过程让我很开心。

比如说：

- 某个 JUCE API 怎么用
- 某个功能的设计思路对不对
- 代码哪里可能有 bug
- 帮我写一些模板性质的代码

我给项目写了 AGENTS.md，定义了一些开发规范和边界。AI Agent 按照这个规范来辅助，不至于跑偏。

但 AI 不是万能的。架构决策、核心逻辑、技术选型这些，还是得自己把关。AI 给的建议有时候会有问题，需要自己判断。AI 的作用是提高效率，不是代替思考。

## 我踩过的坑

做这个项目还是踩了一些坑，记录一下。

**音频设备初始化顺序问题**。刚做出来的时候，有时候启动之后第一个音会走音。后来定位到是音频设备初始化的顺序问题，加上 25ms warmup 才解决。

**MIDI 导入播放首音无声**。导入一个 MIDI 文件之后播放，第一个音经常没声音。这个问题花了不少时间排查，最后加了 playback-start pre-roll 才修好。

**散落的 Logger 调用**。代码里东一个 `juce::Logger::writeToLog`，西一个 `DBG`，调试的时候很乱。后来统一成了 `DP_LOG_*` 系列宏，Debug 和 Release 行为也分开了。

**MainComponent 膨胀**。这个前面说过了，就是堆代码堆出来的。

## 当前状态

devpiano 现在是什么状态？

v0.1.1 是一个初步可用的版本。核心的演奏、录制、回放、导出功能都有了。可以保存和打开演奏文件，可以调播放速度。工程基础设施（Diagnostics、测试夹具）也做了一些。

还没做的：

- 最近文件列表和拖拽打开
- 基础 MIDI 编辑
- VST3 插件离线渲染
- 外部 MIDI 硬件验证（手头没有设备）

这些都是计划中的功能，但现在先不做，保持项目在一个"可用但精简"的状态。

## 接下来

短期想做的：

- 最近文件列表和拖拽打开
- 基础 MIDI 编辑（删除音符）

长期的话，有多轨支持、Piano roll 编辑器、多语言 UI 这些想法。但这些都是比较远的事情，先把手头的做好。

## 总结

回头看这段时间，devpiano 带给我的不只是"做出一个能用的工具"。

首先是对 AI 发展的震撼。自己能够与 AI 配合做项目，从想法到实现，AI 能够理解我的意图、帮我把项目做得越来越标准化，这种体验是以前没有过的。

然后是技术上的收获。钢琴、音乐、编程，这三个东西加在一起本身就是一件很开心的事。

最后是学会了如何做项目。AI 工具、Agent 工具、项目管理、如何标准化、工程化，这些东西以前只是听说过，真正做起来才发现里面有很多门道。

这是一个从 0 到 1 完整做一个项目的体验：从想法到代码仓库，从第一个功能到第一个发布版本，从堆代码到考虑可维护性。这个过程学到的东西比单纯写代码要多。

项目还远没有完成，但至少它证明了一件事：我可以把一个想法，从脑子里搬到一个能运行的代码仓库。

剩下的，慢慢来。
