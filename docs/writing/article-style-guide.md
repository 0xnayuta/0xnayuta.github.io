# 文章风格指南

本文档定义博客文章的个人风格，适用于项目复盘、开发笔记等类型文章。

## 核心原则

### 应该

- 中文为主，简洁表达
- 保持个人视角，第一人称叙述
- 技术细节准确，有代码有真相
- 像和朋友分享经验一样写文章
- 直接说结论，少绕弯子
- 遇到问题就写问题，不刻意美化
- 适合写 devpiano、devkit-pi、pi-lsp、pi-subagents、pi-slim-agents 等个人项目

### 不应该

- 营销腔：不要"本文将""综上所述""在当今时代""助力数字化转型"
- 模板腔：不要每段都用"首先...其次...最后..."硬套
- 夸大其词：不要"最优秀""最强""完美"
- 编造数据：star 数、下载量等必须查证
- 把计划写成事实：未实现的功能不能写成已完成

## 文章语气

### 推荐风格

```
我最近在做一个 pi 的插件系统...
这个功能花了我两天时间才调通...
踩了一个坑，记录一下...
说实话，这个设计我现在也不满意...
```

### 避免风格

```
本文将深入探讨 pi 插件系统的设计与实现...
综上所述，插件系统的优势在于...
在当今时代，开发者工具的重要性不言而喻...
```

## 内容结构建议

### 项目复盘类文章

按时间线或问题线组织：

1. 背景：为什么做这个
2. 目标：想解决什么问题
3. 实现：怎么做的
4. 困难：遇到了什么问题
5. 解决：怎么解决的
6. 反思：现在看有什么不足
7. 展望：后续计划

### 技术教程类文章

1. 场景：什么情况下需要
2. 方案：有哪些选择
3. 推荐：我的选择和理由
4. 示例：核心代码
5. 注意事项：容易踩的坑

## 格式规范

### 标题

- 清晰表达主题
- 可以带项目名，如「pi-subagents 发布记」
- 避免标题党

### 代码块

- 必须指定语言：` ```typescript `
- 长代码要有关键注释
- 避免大段无注释的代码

### 列表

- 有序列表用于步骤
- 无序列表用于并列项
- 列表项不要太长

### 链接

- 外部链接要验证有效性
- GitHub 链接使用完整地址
- 图片使用可靠图床

## 个人项目复盘的特殊要求

### 必须包含

- 项目背景和动机
- 技术选型理由
- 核心设计思路
- 遇到的实际问题
- 当前的局限性

### 可以省略

- 完整的项目介绍（README 更适合）
- 与竞品的详细对比
- 过度的自我评价

### 态度

- 可以承认不足："这个设计有点丑，但先跑起来"
- 可以记录纠结："我纠结了很久要不要加这个功能"
- 可以写教训："如果重来，我会..."

## 禁止事项

1. 不确定的信息不要写，已注明来源的除外
2. 不要把别人的成果说成自己的
3. 不要在技术文章里夹带私货
4. 不要为了字数凑内容
